以 **goal-driven execution** 的方式验证当前工作目标是否真正达成 —— 不是"看起来做完了"，而是"逐项跑过、有输出支撑"。

目标参数：$ARGUMENTS

## 执行步骤

1. **确定成功标准**
   - 若仓库存在 `scripts/verify-goal.sh`，它就是机械化的成功标准 —— 直接 `bash scripts/verify-goal.sh`，把它的每一项当作一条 criterion。
   - 否则，把 `$ARGUMENTS`（或当前会话的任务上下文）拆成一组**明确、可执行**的验证项：build / test / lint / 契约 / 关键产物存在性。

2. **执行验证**：逐项跑，记录每项 ✅ 通过 / ❌ 失败，失败的附上真实命令输出（不要转述、不要脑补结果）。

3. **未达成则修复**：对每个 ❌ 定位根因 → 最小化修复 → 重跑该项确认转绿。不跳过、不把失败说成通过；破坏性修复前先确认。

4. **Loop until green**：重复 2–3，直到全部 ✅，或遇到需要用户拍板的阻塞 —— 那就停下，说清卡在哪、为什么、需要什么决定。

5. **报告**：列出每项最终状态 + 一句总结。全部通过才写 `🎉 GOAL 达成`；否则 `⚠️ N 项未达成` 并列出。

## 纪律

- **实测核对，不脑补**：每个"通过"必须有实际命令输出支撑。
- **如实报告**：失败附输出；跳过 / 降级的项明确标注，不掩盖。
- **范围克制**：只验证与修复目标相关项，不顺手改无关代码。
